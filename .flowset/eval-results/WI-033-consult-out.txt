제안한 방향에 동의합니다. 구현 전 결정안으로는 아래처럼 좁히는 게 가장 낫습니다.

Q1. `src/constants/dashboard-copy.ts` 단일 파일을 권장합니다.  
대시보드 전체 한글화는 화면, CSV, 버튼, 빈상태, 테이블 헤더가 서로 같은 용어를 공유해야 하므로 co-location보다 단일 SoT가 유리합니다. 기존 `src/constants/navigation.ts`, `game-constants.ts`처럼 flat `constants/`에 두는 것도 현재 프로젝트 컨벤션과 맞습니다.

구조는 중첩 객체가 적절합니다. 다만 export는 여러 개보다 하나의 named export가 더 깔끔합니다.

```ts
export const DASHBOARD_COPY = {
  COMMON: { loading: "불러오는 중...", unknown: "알 수 없음" },
  MEMBERS: { title: "멤버", table: { name: "이름" } },
  ...
} as const;
```

키는 영문, 값은 한국어가 맞습니다. 키까지 한글화하면 검색성, refactor 안정성, 타입 추론이 떨어집니다.

Q2. scope는 dashboard로 엄격 고정하는 쪽에 동의합니다.  
이번 WI의 목표는 “어드민 대시보드 한글화 + copy constants 분리”이므로 `src/app/dashboard/**`, `src/components/dashboard/**`까지만 처리하는 게 맞습니다. `player-list.tsx`의 `"No other players"` 같은 인-스페이스 문자열은 제품 전체 한글화 이슈로 연결되기 쉬워서 범위가 흐려집니다. 이번에 포함해야 할 강한 이유는 없어 보입니다.

Q3. CSV 헤더, 에러 메시지, select 라벨, placeholder, aria-label, title 모두 포함하는 데 동의합니다.  
특히 `aria-label`과 `placeholder`는 사용자 노출 문자열로 취급해야 하고, CSV는 Excel에서 직접 보이는 산출물이므로 한국어 제품에서는 한글 헤더가 자연스럽습니다.

다만 CSV에서 enum 값까지 번역할지는 의도적으로 정해야 합니다. 제 권고는 다음입니다.

- CSV 헤더: 한글화
- `Yes` / `No`, `Unknown`, `[unserializable]`: 한글화
- role/restriction/eventType 같은 서버 enum 값: 화면 표시용은 한글 라벨로 매핑하되, CSV에서는 “사용자용 다운로드”라면 한글 라벨로 통일
- 감사/운영 로그 성격상 원본 코드가 중요하면 컬럼명을 `이벤트 유형`으로 두고 값은 `ENTER`처럼 원본 코드 유지

이 WI가 “한국어 단일 언어 제품” 정합성 목적이라면 CSV 값도 가능한 한 한글 표시 라벨로 맞추는 편이 더 일관됩니다. 단, payload JSON 내부 키는 데이터라서 번역하지 않는 게 좋습니다.

Q4. 동적/보간 문자열은 함수 엔트리가 적절합니다.  
`memberCount(visible, total)`, `spotlightGrantCount(count)`, `loadedCsvCount(count)`처럼 copy 객체 안에 함수로 두는 방식이 가장 단순합니다. 별도 formatter 유틸까지 만들 필요는 없습니다.

예:

```ts
MEMBERS: {
  count: (visible: number, total: number) =>
    visible === total ? `${visible}명` : `${visible} / ${total}명`,
}
```

주의할 점은 함수가 React 컴포넌트를 반환하지 않고 문자열만 반환하게 하는 것입니다. 그러면 테스트와 재사용이 쉽습니다.

Q5. 가장 중요한 위험은 “표시 라벨”과 “도메인 값/API 값”을 섞는 회귀입니다.  
현재 `OWNER`, `STAFF`, `NONE`, `MUTED`, `ENTER`, `MESSAGE` 같은 값이 UI에 그대로 노출되어 있지만, 이 값들은 필터 value, API payload, 서버 enum, CSV 데이터로도 쓰입니다. 구현자가 한글화를 하면서 `<option value="OWNER">`를 `<option value="소유자">`처럼 바꾸면 필터/API가 깨질 수 있습니다.

따라서 상수 파일에는 반드시 둘을 분리하는 구조가 필요합니다.

```ts
roleLabels: {
  OWNER: "소유자",
  STAFF: "스태프",
  PARTICIPANT: "참여자",
}
```

렌더링은 `roleLabels[m.role]`, 전송/필터 값은 계속 `m.role` 또는 `"OWNER"`를 써야 합니다. 이 원칙만 지키면 제안한 설계는 충분히 안정적입니다.
