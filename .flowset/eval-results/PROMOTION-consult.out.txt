Read-only로 핵심 파일을 대조했고, 네 전제에 대체로 동의합니다. 단, 몇 군데는 “조건부 동의”입니다.

**A. 시퀀싱**

동의: `env 선확정 → DB 선처리 → merge → smoke`가 맞습니다.

마이그레이션은 분리 가능하지만 둘을 나눠 봐야 합니다. WI-013 인덱스는 런타임 필수 의존이 아니므로 merge 전/후 모두 가능하고, 운영 부하가 낮은 시간에 독립 적용해도 됩니다. WI-016은 `isShared` 컬럼이 prod에 실제로 없으면 코드 배포 후 Prisma 쿼리가 깨질 수 있으므로, “이미 존재”를 `\d "GeneratedAsset"`로 확정하기 전에는 merge 전에 처리해야 합니다.

즉, 컬럼 존재 확인은 go-live gate이고, 인덱스는 성능/운영 gate입니다.

**B. AUTH_SECRET 리스크**

가장 안전한 검증은 “값을 보지 않고 fingerprint를 비교”하는 방식입니다. Vercel production env와 OCI `.env` 각각에서 `AUTH_SECRET`의 길이와 SHA-256 fingerprint 앞자리만 출력해 비교하세요. secret 원문을 채팅이나 로그에 남기면 안 됩니다.

기존 값이 같은지 보장할 수 없고 Vercel 값도 열람 불가라면, 안전한 선택지는 둘 중 하나입니다.

1. Vercel prod `AUTH_SECRET`을 확인할 수 있으면 그 값을 OCI에 맞춘다.
2. 확인할 수 없으면 새 32자 이상 secret을 생성해 Vercel과 OCI 양쪽에 같은 값으로 재설정한다.

단, `AUTH_SECRET`이 NextAuth 세션 서명에도 쓰이고 있다면 새 값으로 바꾸는 순간 기존 로그인 세션이 무효화될 수 있습니다. 장애는 아니지만 사용자 로그아웃 영향입니다. 이 가능성 때문에 “새 값 강제 통일”은 유지보수 창에서 하는 게 맞습니다.

dry-run은 develop 코드로 Vercel preview/staging과 OCI 임시 socket 컨테이너를 띄워 같은 secret으로 `/api/socket/token → socket auth`를 실제 통과시키는 방식이 제일 좋습니다. production merge 없이 양쪽 코드 경로를 검증할 수 있습니다.

**C. 롤백**

부분 동의입니다. Vercel instant rollback + main revert는 웹 롤백으로는 충분합니다. 하지만 OCI는 `docker compose up --build -d`가 새 이미지를 빌드하고 컨테이너를 교체하는 구조라, “코드 revert 후 재배포”만으로는 배포 도중 실패나 새 컨테이너 crash-loop에 대한 즉시 회귀가 약합니다.

보강하세요: 승격 전 현재 OCI commit SHA, image ID, container ID, `.env` checksum을 기록하고, 가능하면 이전 이미지를 명시 태그로 보존하세요. 배포는 `build → up → healthcheck → logs 확인` 순서로 보고, 실패 시 이전 이미지/이전 commit으로 즉시 `up -d` 가능한 절차가 있어야 합니다.

**D. 마이그레이션 안전성**

WI-013의 일반 `CREATE INDEX` / `DROP INDEX`는 prod 대용량에서 위험합니다. 특히 non-concurrent index 작업은 쓰기 차단, long transaction 대기, lock queue를 만들 수 있습니다.

일반 `migrate deploy`가 무방한 전제는 꽤 좁습니다: `Space`가 작고, 트래픽 낮은 시간이며, long transaction이 없고, 작업 시간이 수 초 이내이며, `lock_timeout`/`statement_timeout`과 즉시 중단 권한이 준비된 경우입니다. 이 전제를 숫자로 확인하지 못하면 direct connection에서 `CONCURRENTLY`로 수동 적용하고 `migrate resolve --applied`가 맞습니다.

**E. 89커밋 일괄 vs 분할**

원칙적으로는 분할 승격이 더 안전합니다. 특히 인증 fail-closed, socket protocol, DB 변경이 동시에 처음 라이브되는 건 빅뱅 리스크입니다.

다만 지금 상태에서 89커밋을 뒤늦게 cherry-pick 분할하면 호환성 검증 비용과 실수 가능성도 큽니다. 이미 WI 단위 통합 테스트가 충분하지 않다면, 임의 분할보다 “강한 gate를 둔 단일 승격”이 더 현실적입니다. 조건은 명확합니다: AUTH_SECRET 검증 완료, 컬럼 존재 확정, OCI rollback 보강, maintenance window, smoke checklist 준비. 이 조건 없이는 빅뱅 반대입니다.

**F. 놓친 위험 1가지**

Vercel 배포 인가 계정 리스크입니다. develop HEAD 작성자는 인가 계정이지만, main에 develop을 머지할 때 생기는 최종 merge commit/squash commit 작성자가 다른 계정이면 Vercel prod 배포가 실패할 수 있습니다. main에 빈 커밋이 하나 있어 단순 fast-forward도 아닙니다.

승격 방식은 반드시 “최종 main HEAD commit author가 Vercel 인가 계정”이 되도록 정해야 합니다. PR merge 버튼, squash merge, rebase merge 각각 author가 어떻게 찍히는지 확인하고, 필요하면 인가 계정으로 merge commit을 만들거나 해당 계정을 Vercel에 인가하세요.
