## 검증 관점 (적대적으로)
1. **인덱스 정확성**: `@@index([status, updatedAt(sort: Desc), id(sort: Desc)])`가 핫 쿼리의 `WHERE status=ACTIVE` + `ORDER BY updatedAt DESC, id DESC`를 실제로 지원하는 컬럼 순서/정렬방향인가? (equality 컬럼 status 선두 → sort 컬럼 순서·방향 일치?) 마이그레이션 SQL이 schema와 일치하는가(컬럼·방향·인덱스명)?
2. **D4 제거 안전성(회귀)**: `@@index([status])` 제거가 다른 쿼리를 깨뜨리는가? 코드베이스에 status 단독 필터/정렬/집계 쿼리(복합 선두 컬럼이 커버 못 하는 패턴)가 있는가? `git grep`으로 `prisma.space` status 사용 전수 확인 권장.
3. **마이그레이션 안전/형식**: 마이그레이션 디렉토리·SQL 형식이 0_init 선례와 정합한가? `DROP INDEX`가 인덱스명(`Space_status_idx`)을 정확히 가리키는가(0_init이 만든 이름)? `prisma/migrations/`가 gitignore인데 force-add됨 — 0_init과 동일 처리로 타당한가, 빠진 산출물(migration_lock.toml 등)이 문제인가?
4. **회귀 없음**: 인덱스 추가/제거는 client 타입·런타임 동작 무변(기계게이트 5/5). 응답/쿼리 결과 동일.
5. **prod 게이트 정확성**: D5(CONCURRENTLY 트랜잭션 불가)·cursor seek 위험·EXPLAIN 필요가 스펙/마이그레이션 주석에 정확히 문서화됐는가? 잘못된 운영 지침은 없는가?
