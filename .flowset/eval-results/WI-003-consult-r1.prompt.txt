## 핵심 제약 (검증 완료)
- 서버는 esbuild로 server/index.ts 단일 번들. `socket/index.ts`/`chat/index.ts` 배럴은 useSocket/useChat 등 React 훅을 함께 export → 서버가 배럴 import하면 React+클라 트리 전체가 번들에 끌려와 빌드 깨짐. 그래서 Dockerfile이 순수 계약 파일 2개만 COPY. **즉 서버 케이스는 배럴 routing으로 해결 불가.**
- socket/internal/types.ts = import 0개 순수 타입. chat/internal/chat-constants.ts = "서버/클라 공통, React 의존성 없음" 명시된 순수 const.
- game-manager.ts가 MainScene을 `await import()` lazy 로드 → game/index.ts는 editor/index.ts를 정적으로 끌지 않음. 따라서 editor→game 배럴 routing 시 editor/index↔game/index 정적 순환은 발생하지 않음(main-scene→editor는 런타임 lazy라 양 배럴 초기화 후 실행).
- RECONNECTION_*/MOVE_THROTTLE_MS는 소켓 상수인데 물리적으로 chat-constants.ts에 위치(오배치 냄새). chat/index.ts는 이들을 현재 export 안 함(DEFAULT_NICKNAME만 export).

## 내 제안
- A(인앱): 배럴 routing으로 해소. game/index.ts에 `TILE_INDEX, extractDefaultMapData, type TilemapResult` 추가 export, editor/index.ts에 `EditorSystem` 추가 export, chat/index.ts에 `MOVE_THROTTLE_MS, RECONNECTION_*` 추가 export. 7개 import 사이트를 배럴로 교체. 정적 순환 없음(위 검증).
- B(서버): 두 순수 계약 파일을 communication 도메인 공유 계약 모듈로 승격하는 게 정석이나(예: `src/features/space/protocol/index.ts` 순수, server가 이 배럴 상대경로 import, Dockerfile/워크플로우 경로 갱신), 신규 모듈+파일 이동+server 8파일+Docker+CI 변경이라 범위/배포 리스크 큼. 대안: 서버는 별도 빌드/프로세스이므로 "순수 계약 파일에 한해 cross-process import 허용"을 명시 예외로 문서화하고 인앱(A)만 정리.
- C(scripts): 이번 WI 제외, 문서화.
