결정: **A를 다음 WI-023으로 먼저 처리**하세요.

근거는 명확합니다. A는 CWE-209 보안 이슈이고, 실표면이 `assets` 1곳이 아니라 26파일 37곳의 API 전반입니다. 500은 정상 플로우가 아니어도 DB 장애, Prisma 예외, 배포 미스매치 때 실제로 노출될 수 있습니다. B는 UX 방어, C는 운영 정책 변경입니다. 둘 다 중요하지만 A보다 우선순위가 낮습니다.

**WI-023 스코프**
- **26파일 37곳 일괄 수정**이 맞습니다. `assets`만 고치면 같은 취약 패턴이 대부분 남아 “해결됨”이라고 말할 수 없습니다.
- 중앙 헬퍼 도입이 맞습니다. 예: `internalServerErrorResponse(error, "Failed to fetch assets", { route: "GET /api/assets" })`.
- 클라이언트 응답은 기존 계약을 최대한 유지해서 `{ error: "Failed to ..." }`만 반환하세요. `details` 제거. `code`는 app.md 불변식상 허용되지만, 500 전체에 새로 추가하지 않는 편이 무회귀성이 높습니다.
- 서버 로그는 유지해야 합니다. `console.error("[api/...]", error)` 또는 헬퍼 내부 구조화 로그로 원본 Error/Prisma error를 서버에만 남깁니다.
- **dev에서도 details를 응답하지 않는 쪽**을 권합니다. 환경별 응답 shape가 갈라지면 테스트와 클라이언트 계약이 흐려집니다. 디버깅은 서버 로그로 충분합니다.
- Prisma 에러는 절대 클라이언트에 원문 반환하지 않습니다. 제약명, 컬럼명, DB URL 일부, 파일 경로가 섞일 수 있습니다.
- 기존 테스트가 `body.error`만 확인하는 곳은 유지됩니다. 다만 `details` 존재를 기대하는 테스트가 있다면 그 테스트가 잘못된 보안 계약이므로 “없어야 한다”로 바꿔야 합니다.
- 검증은 `rg "details: error instanceof Error"`가 0건이어야 하고, 대표 라우트 2~3곳은 500에서 `details` 미포함을 테스트로 고정하세요.

**offset 상한은 이번 WI에서 분리**하세요.  
`parsePageNumber("100000") === 100000`이 현재 테스트로 박혀 있고, 소비처 실표면도 낮습니다. A는 보안 노출 제거로 닫고, offset cap은 별도 WI로 `MAX_PAGE_NUMBER` 정책을 정한 뒤 처리하는 게 맞습니다.

B는 A 이후 후보입니다. retry를 넣더라도 401/403 같은 비-일시 장애는 즉시 throw, 네트워크 오류/408/429/5xx만 2~3회 짧은 exponential backoff가 적절합니다.

C는 지금 하지 마세요. 코드 주석상 graceful degrade가 의도된 운영 정책입니다. fail-fast는 안전망이 아니라 배포 기동 실패로 바뀌는 정책 변경이라 별도 승인 게이트가 필요합니다.

놓친 위험 1가지: **500 응답의 `error` 문자열 자체도 라우트별 내부 동작을 드러낼 수 있습니다.** 이번에는 회귀 최소화를 위해 기존 `"Failed to ..."` 문구를 유지하되, 장기적으로는 외부 공개 API와 관리자 API의 에러 문구 노출 수준을 따로 정리하는 후속 보안 WI가 필요합니다.
